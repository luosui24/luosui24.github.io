## 一、概述
以下全权由ai辅助分析

IDA Pro 使用 1024 位 RSA 对许可证文件 (idapro.hexlic) 进行签名验证。
公钥硬编码在二进制文件的 .rdata 段中，且未被代码完整性保护。

攻击者只需修改二进制中公钥的一个字节（0x5C → 0xCB），
替换为已知对应私钥的新模数，即可为任意许可证数据生成有效签名。

## 二、关键函数分析

### 2.1 许可证管理器构造函数 (new_license_manager)

- 分配 552 字节堆内存
- 设置虚函数表指针 vtable
- 初始化各成员为默认值:

| 偏移 | 含义 |
| :--- | :--- |
| +0x000 | vtable 指针 |
| +0x008~+0x188 | 基类和其他子对象 (384 字节) |
| +0x188~+0x1F0 | 许可证数据指针区 |
| +0x1B0 | qmutex_create() — 线程互斥锁 |
| +0x1F8 | max_retries = 10 |
| +0x1FC | check_interval_secs = 7200 (2 小时) |
| +0x200 | offline_mode = false |
| +0x220 | is_activated = false |

### 2.2 单例工厂 + 初始化 (get_license_manager)

全局变量:
- qword_10206F888 — 就绪标志（必须非空）
- qword_10206F890 — license_manager* 单例

虚函数表:

| 索引 | 偏移 | 用途 |
| :--- | :--- | :--- |
| 0 | +0x00 | 析构函数 |
| 1 | +0x08 | add_license_path() — 注册许可证文件路径 |
| 2 | +0x10 | initialize() — 加载验证参数（含公钥） |
| 3 | +0x18 | acquire_license() — 核心验证: 加载并验证许可证 |
| 4 | +0x20 | ? |
| 5 | +0x28 | configure() — 运行时参数配置 |
| 6~8 | +0x30~+0x40 | ? |
| 9 | +0x48 | checkin_license() — 归还浮动许可证 |
| 10 | +0x50 | get_loaded_license() — 查询已加载的许可证 |
| 11 | +0x58 | verify_expiry() — 验证有效期 + 签名 |

执行流程:
assert(就绪标志) → 检查单例 → new_license_manager()
→ vtable[2].initialize() → vtable[5].configure()
→ 扫描子目录 → 对每个路径调用 add_license_path()

### 2.3 子目录扫描器 (get_ida_subdirs)

参数:
- a1: 输出目标结构体
- a2: 子目录名 ("sig", "ids", "idc", "til" 等)
- a3: 标志位掩码 (位 0=需处理器名, 位 1=标准路径优先)

收集的目录来源:
1. 标准目录: idadir(subdir)
2. 用户自定义目录列表
3. 环境变量覆盖 (IDASGN / IDAIDS / IDAIDC / IDATIL)

### 2.4 许可证加载核心 (sub_10005C740)

流程:
1. 初始化上下文结构体（清零 232 字节 + 设置参数）
2. 初始化许可证子系统 + atexit 注册
3. get_license_manager() 获取单例
4. vtable[10].get_loaded_license() 检查已有许可证
5. 无效 → vtable[3].acquire_license()
   - 创建请求/响应对象
   - 调用验证函数
   - 失败时弹 UI + 日志

### 2.5 打开文件时的许可证守卫 (sub_10005EAB0)

流程:
1. 打开/创建 IDB 数据库文件 (sub_10005F200)
2. 需要许可证验证时:
   → vtable[11].verify_expiry(current_time, &error_code)
3. 结果处理:
   - 验证通过 → 允许打开
   - 错误码 1~5 → 宽限（临时错误）
   - 错误码 >=6 → 拒绝:
     "Your license has expired, you cannot open new file."

### 2.6 关闭数据库时的许可证归还 (sub_1000AA9F0)

全局变量:
- qword_10206F8D8: 互斥锁
- qword_10206F8E0: 数据库上下文指针数组
- qword_10206F8E8: 数组长度

流程:
1. 从全局数组中移除当前上下文
2. 从尾部扫描最后一个活跃上下文
3. 还有其他活跃数据库 → 仅析构，切换到其他库
4. 最后一个数据库关闭 → vtable[9].checkin_license() 归还

### 2.7 核心 RSA 验签函数 (sub_1003F32B0)

JSON 条目布局（每项 40 字节）:

| 偏移 | 含义 |
| :--- | :--- |
| +0 | key 指针 |
| +8 | value 指针 |
| +16 | 额外数据 |
| +24 | 类型标志 (2=字符串) |
| +32 | 长度/数据指针 |

验证流程:
1. 搜索 JSON 中的 "signature" 键
2. 计算 payload 的 SHA-256 (sub_1003F3EF0)
3. 解析十六进制签名为 128 字节
4. RSA 公钥解密:
v25 = sub_1003F9910(v41, &qword_101990120, &qword_101990080)
(输出) (指数 e=0x13) (模数 n)
5. 检查偏移 64+ 区域无非法非零字节
6. 提取偏移 64~95 的哈希 vs 实际 SHA-256 比较

关键发现 — 签名格式:
RSA 解密后的 128 字节:

│ 前 64 字节 │ SHA-256 (32 字节) │ 全 0 │ 忽略 │ (偏移 64~95) │ (96~127) │


## 三、RSA 签名验证协议

### 3.1 验签流程

1. 从 JSON 提取 payload
2. 按字母序序列化: json.dumps({"payload": ...}, sort_keys=True)
3. 计算 SHA-256(payload_json) → 32 字节
4. RSA 公钥解密签名: decrypted = signature^e mod n
5. 从 decrypted[64:96] 提取哈希
6. 比较与实际计算的 SHA-256 是否一致
7. 检查 decrypted[96:128] 全零

### 3.2 密码参数

- 算法: RSA-1024
- 指数 e: 0x13 (= 19)
- 模数 n: 128 字节 (见第四节)

## 四、硬编码公钥定位

### 4.1 模数 n 的位置

- 段: .rdata
- 地址: 0x101990080 (基地址相关)
- 符号: qword_101990080
- 数据: dq 6E5478F95C42FDEDh (小端)

完整 128 字节以小端格式连续存储。
从偏移 +4 处找到 0x5C，修改为 0xCB 即可替换公钥。

### 4.2 指数 e 的位置

- 地址: 0x101990120
- 符号: qword_101990120

### 4.3 交叉引用
sub_1003F32B0 +0x1DE:
v25 = sub_1003F9910(v41, &qword_101990120, &qword_101990080);

## 五、攻击链全景

攻击者操作:
1. 修补 IDA 二进制: 搜索 edfd42_5c_f978 → edfd42_cb_f978
   (公钥 n → n', 使得私钥 d' 已知)
2. 生成 idapro.hexlic:
   - 构造 JSON payload
   - 计算 SHA-256(payload)
   - RSA 签名: signature = (填充 + 哈希 + 零填充)^d' mod n'
   - 输出到 idapro.hexlic
3. 放置到 IDA 任意扫描目录

IDA 内部执行:
1. 启动 → get_license_manager()
   → initialize() 加载 n' (已被替换)
   → add_license_path() 扫描到 idapro.hexlic
2. 打开文件 → verify_expiry()
   → RSA 解密: 用修补后的 n' 验签 → 通过
3. 正常使用 IDA Pro 所有功能

## 六、安全设计缺陷总结

| 缺陷 | 风险等级 |
| :--- | :--- |
| 公钥硬编码在二进制 | 🔴 严重 |
| 无代码完整性校验 | 🔴 严重 |
| 仅 1 字节特征可定位 | 🟠 高 |
| 1024 位 RSA | 🟡 中 (但非被攻破的原因) |
| 错误码宽限策略 1~5 都允许 | 🟢 低 |

核心问题:
验证链: 二进制完整性 → 公钥可信 → RSA 验签 → 许可证有效
            ✗        未保障     被替换      形同虚设   永远通过


如果我是防御方:
1. 公钥不硬编码（远程下发或嵌入代码签名证书）
2. 启动时自校验自身 SHA-256
3. 公钥分散存储，运行时拼接
4. 定期轮换公钥 + 强制更新
5. 公钥存储 HMAC 或数字签名，运行前验证完整性
